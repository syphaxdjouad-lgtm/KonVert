## Périmètre

Cet audit couvre les flows fonctionnels E2E non traités par l'audit v2 (qui se concentrait sur corrections de bugs B1-B8, sécurité, SEO, perf). L'audit v2 n'a pas audité : les edge cases wizard, les race conditions UI, les flux d'annulation Stripe, les cas i18n, la gestion des erreurs côté utilisateur, la couverture a11y fonctionnelle, ni les bugs d'état intermédiaire (loading, error, empty states).

Méthode : analyse statique du code source (`src/app`, `src/app/api`). Aucun test Playwright exécuté — recommandations E2E listées en section finale.

## P0 — BLOQUANTS (interdisent le lancement ou la conversion)

### P0-1 | Flow Paiement > Annual toggle silencieux
**Flow :** `/pricing` → checkout Stripe
**Scénario :** L'utilisateur sélectionne la facturation annuelle (toggle UI "Annuel / Mensuel"), puis clique "Démarrer avec Pro".
**Observation :** `handleCheckout(plan.id)` envoie uniquement `{ plan }` à `/api/stripe/checkout`. La variable `annual` n'est **jamais transmise**. `STRIPE_PRICES` ne contient que 3 price IDs mensuels (`STRIPE_PRICE_STARTER`, `STRIPE_PRICE_PRO`, `STRIPE_PRICE_AGENCY`). Il n'existe pas de `STRIPE_PRICE_STARTER_ANNUAL` côté API.
**Résultat observé :** L'utilisateur pense souscrire à l'annuel (-20%), il est débité au tarif mensuel. Stripe crée une subscription mensuelle. Fraude commerciale involontaire.
**Fix :** Passer `{ plan, annual }` dans le body du fetch, ajouter `STRIPE_PRICE_*_ANNUAL` dans `lib/stripe/index.ts`, créer les price IDs correspondants dans Stripe Dashboard, ou désactiver le toggle annuel jusqu'à implémentation.

### P0-2 | Flow Signup > Redirect dashboard sans confirmation email
**Flow :** `/signup` → dashboard
**Scénario :** Un user s'inscrit avec un email valide.
**Observation :** `handleSignup` appelle `supabase.auth.signUp()` puis fait immédiatement `router.push('/dashboard')` (ligne 71). Supabase en mode "Email Confirmation Required" (comportement par défaut) envoie un email de confirmation mais la session n'est pas encore active. L'user arrive sur `/dashboard` avec une session non confirmée. Selon la config Supabase, le middleware `updateSession` peut le laisser passer (si email confirm est désactivé en bêta) ou le rediriger en boucle vers `/login`.
**Résultat observé :** Deux bugs selon config : (a) si confirmation désactivée, OK fonctionnellement mais aucun email de confirmation envoyé et l'email de bienvenue (`/api/email/welcome`) part même si l'user est bot/faux email. (b) si confirmation activée, l'user arrive sur `/dashboard`, voit un état vide ou est redirect login, ne comprend pas qu'il faut confirmer.
**Fix :** Après `signUp`, afficher un écran intermédiaire "Vérifie ta boîte email" au lieu de push direct vers dashboard. Conditionner l'email de bienvenue à la confirmation effective (via webhook Supabase Auth ou trigger DB).

### P0-3 | Flow Checkout > Pas de toast/feedback erreur visible sur pricing
**Flow :** `/pricing` → checkout Stripe
**Scénario :** La session Stripe échoue (Stripe down, clé invalide, réseau coupé).
**Observation :** `handleCheckout` catch l'erreur et fait `console.error(err)` + `setLoading(null)` (ligne 167). Il n'existe **aucun setState pour un message d'erreur** visible dans ce composant. L'UI revient à l'état initial sans aucun feedback.
**Résultat observé :** L'user clique, voit un spinner, puis rien. Il croit que le bouton est cassé. Taux d'abandon direct.
**Fix :** Ajouter `const [checkoutError, setCheckoutError] = useState<string | null>(null)` et afficher le message sous le CTA cliqué.

---

### P0-4 | Flow Quota > decrement_quota silencieux en cas d'échec rollback
**Flow :** `/api/generate` (dashboard wizard)
**Scénario :** DeepSeek timeout après incrément quota réussi. `rollbackQuota()` est appelé mais la RPC `decrement_quota` échoue (Supabase down, RLS, timeout).
**Observation :** L'erreur est loguée `console.error('[/api/generate] decrement_quota échoué:', rbErr)` (ligne 48) mais aucun mécanisme de retry ni d'alerte Sentry n'est déclenché. Le quota est définitivement brûlé pour une génération qui n'a rien produit.
**Résultat observé :** L'utilisateur perd une génération de son quota mensuel sans obtenir de page. Sur un plan Starter (75 pages/mois), répété 3 fois ça représente 4% du quota disparu.
**Fix :** Ajouter `Sentry.captureException(rbErr)` dans le bloc catch du rollback. Envisager une table `quota_rollback_queue` pour rejouer les decrements manqués.

## P1 — MAJEURS (dégradent significativement l'expérience ou la conversion)

### P1-1 | Flow Essai > Pas de protection double-soumission (race condition UI)
**Flow :** `/essai` → `/api/generate/public`
**Scénario :** L'utilisateur clique deux fois rapidement sur "Générer ma page en 30 secondes" avant que le state passe à `'generating'`.
**Observation :** Le bouton submit n'a pas `disabled` pendant l'envoi. La protection contre la double requête est côté serveur (unique index `public_previews_email_active_uniq`), ce qui retourne un 409. La gestion 409 côté client (`essai/page.tsx` ligne 271-275) redirige vers la preview existante. Mais entre le 1er clic et le changement de state React (`setStep('generating')`), il y a une fenêtre de ~50-100ms pendant laquelle un 2e clic déclenche une 2e requête. La 2e requête paie le coût du scraping + DeepSeek avant d'échouer sur l'index unique.
**Fix :** Ajouter un `useRef` isSubmitting côté client : `if (isSubmitting.current) return; isSubmitting.current = true;` avant le fetch, reset dans le finally. Ou `disabled={isGenerating}` dès le premier setState.

### P1-2 | Flow Wizard > Navigation Précédent depuis step 8 efface l'état "generating"
**Flow :** `/dashboard/new` — wizard 8 étapes
**Scénario :** L'user est en step 8 (Récap), clique "Générer". Pendant la génération (`mode === 'generating'`), il appuie sur le bouton retour du navigateur (Back browser).
**Observation :** La page se remonte sur `/dashboard/new`. Le composant est remonté fresh (state perdu). L'URL a été modifiée via `window.history.replaceState` uniquement après le succès autosave — donc pendant la génération il n'y a pas de `page_id` dans l'URL. Le remontage repart de step 1, toutes les données wizard sont perdues. L'utilisateur doit tout recommencer.
**Fix :** `useBeforeUnload` ou `router.beforePopState` (Next.js) pour intercepter la navigation arrière pendant `mode === 'generating'` et afficher une modale de confirmation.

### P1-3 | Flow Wizard > Refresh navigateur pendant génération = perte totale
**Flow :** `/dashboard/new` → mode `'generating'`
**Scénario :** L'user attend 30-60s la génération, son navigateur freeze ou il appuie F5 par réflexe.
**Observation :** Le state React est perdu. Le quota a déjà été incrémenté (`check_and_increment_quota` est appelé avant la génération). L'autosave n'a pas encore eu lieu (il ne se fait qu'après succès de génération). L'user perd 1 page de quota et revoit le wizard vide.
**Fix :** Au démarrage de la génération, persister dans `sessionStorage` les paramètres du wizard (url, style, tone, langue). Si remontage détecté avec `page_id` absent mais `sessionStorage` présent, proposer "Reprendre la génération précédente". Et/ou : déclencher l'autosave draft AVANT de lancer la génération IA (status: 'generating').

### P1-4 | Flow Wizard > Bouton "Retour" en mode éditeur → perte HTML non sauvegardé
**Flow :** `/dashboard/new` → mode `'editor'`
**Scénario :** L'user a généré sa page, est dans l'éditeur, modifie du texte dans le builder, puis clique "Retour" (bouton dans la header de l'éditeur, ligne 573-576).
**Observation :** `onClick={() => setMode('wizard')}` sans confirmation ni sauvegarde préalable. L'HTML modifié est perdu. L'autosave a créé la page en DB mais avec l'HTML non édité.
**Fix :** Tracker un `isDirty` dans le builder, afficher une `window.confirm()` ou une modale "Des modifications non sauvegardées seront perdues".

### P1-5 | Flow Publication > Bouton "Publier" absent si `stores.length === 0` sans explication
**Flow :** `/dashboard/new` → mode éditeur → publish
**Scénario :** L'user génère sa page mais n'a pas encore connecté de store.
**Observation :** La condition `stores.length > 0 && pageId` (ligne 604) cache complètement le bouton Publier. Il n'y a aucun message alternatif, aucun lien vers `/dashboard/stores`. L'user voit juste un bouton absent sans comprendre pourquoi.
**Fix :** Afficher un bouton désactivé "Publier" avec tooltip "Connecte d'abord un store →" ou un lien inline vers `/dashboard/stores`.

### P1-6 | Flow Signup > Email déjà utilisé : message Supabase brut en anglais
**Flow :** `/signup` → form submit
**Scénario :** L'utilisateur entre un email déjà enregistré dans Supabase.
**Observation :** `authError.message` est affiché directement (ligne 50 : `setError(authError.message)`). Supabase retourne "User already registered" en anglais. Dans une interface 100% française, c'est incohérent et peu professionnel.
**Fix :** Mapper les codes d'erreur Supabase : `if (authError.message.includes('already registered')) setError('Cet email est déjà utilisé. Connecte-toi ou réinitialise ton mot de passe.')`.

### P1-7 | Flow Cron emails > Timezone drift possible sur `daysPassed`
**Flow :** `/api/cron/preview-emails` — exécuté à 8h et 9h UTC (vercel.json)
**Scénario :** Un user génère sa page à 23h55 heure de Paris (= 21h55 UTC). Le lendemain à 8h UTC, `daysPassed = Math.floor((now - createdAt) / 86400000)` = `Math.floor(36300000 / 86400000)` = 0 (< 1 jour UTC écoulé). L'email J+1 n'est pas envoyé. Il sera envoyé à 8h UTC J+2 (soit ~34h après la génération au lieu de 24h).
**Observation :** Calcul en millisecondes UTC pur, sans ancrage sur le jour calendaire local de l'utilisateur. Pour des users en UTC+5/UTC+8 (MENA, Asie), le drift peut atteindre 5-8 heures, parfois faisant sauter un email de la séquence si le cron tourne une seule fois par jour.
**Fix :** Utiliser `Math.round` au lieu de `Math.floor` (réduit le drift à ±12h), ou ajouter une marge de ±4h dans la condition : `d <= daysPassed + 0.5`.

### P1-8 | Flow Génération public > `language` non validé côté `/api/generate/public`
**Flow :** `/essai` → `/api/generate/public`
**Scénario :** Un attaquant (ou bug client) envoie `language: 'zh'` dans le body.
**Observation :** `/api/generate/public/route.ts` (ligne 77) transmet `body.language` directement à `generateLandingPage()` sans validation. `generateLandingPage` dans `/lib/anthropic/generate.ts` valide contre `ALLOWED_LANGS` (qui inclut `zh`), donc la génération fonctionne. Mais le template `templateEtecBlue` utilisé pour la page gratuite n'a pas été testé en `zh` (contenu RTL-adjacent, polices Google Fonts sans CJK). Résultat potentiel : page générée avec texte chinois sur une mise en page non adaptée.
**Par contre :** le wizard dashboard `/api/generate` valide aussi via `ALLOWED_LANGS`. La route publique accepte donc `zh` en production mais l'UI `/essai` ne propose que `fr`/`en`. Un utilisateur anglophone qui passe `?lang=zh` en URL obtient une UI EN mais génère en ZH.
**Fix :** Ajouter une validation explicite `if (!['fr','en'].includes(body.language)) body.language = 'fr'` dans `/api/generate/public` puisque cette route ne supporte que ces deux langues en UI. Ou étendre l'UI pour supporter toutes les langues du wizard.

### P1-9 | Flow Preview > iframe `height: calc(100vh - 100px)` cassé sur mobile
**Flow :** `/preview/[id]`
**Scénario :** Utilisateur mobile (iPhone 14, 844px viewport). La sticky top bar prend ~56px, la sticky bottom bar ~72px. L'iframe est à `calc(100vh - 100px)` = 744px. La bottom bar fixée en `position: fixed` chevauchera le bas de l'iframe de ~28px.
**Observation :** Le contenu bas de la landing page générée (souvent le bouton d'achat ou la section FAQ) est partiellement masqué par la barre CTA fixée en bas. Sur mobile Safari, `100vh` inclut la barre d'adresse, aggravant le problème.
**Fix :** `height: calc(100dvh - 100px - 72px)` (dvh = dynamic viewport height, supporté depuis iOS 15.4) ou padding-bottom sur l'iframe.

### P1-10 | Flow A11y > 138 `<button>` pour 13 `aria-label` — ratio 9.4%
**Observation :** Grep : 138 occurrences de `<button` dans le code TSX, 13 `aria-label` au total (tous fichiers confondus). Les boutons sans label textuel visible (icône seule) représentent au minimum : les boutons X de fermeture de modales (stores/page.tsx, new/page.tsx), les boutons de suppression photos dans le wizard, les boutons de navigation du wizard. Ces boutons sont illisibles par les screen readers.
**Exemples critiques :**
- Bouton X fermeture ConfirmModal dans `stores/page.tsx:33` : pas d'`aria-label`
- Boutons suppression photos wizard (new/page.tsx:899) : pas d'`aria-label`
- Bouton X fermeture `partialWarning` (new/page.tsx:662) : a un `aria-label="Fermer"` ✓
- Bouton X fermeture `publishSuccess` (new/page.tsx:697) : a un `aria-label="Fermer"` ✓
- Toggle URL/Manuel dans wizard (new/page.tsx:808) : texte visible, OK
**Fix :** Ajouter `aria-label` sur tous les boutons icon-only. Priorité : boutons X dans modales et boutons de suppression.

## P2 — MINEURS (friction UX ou couverture incomplète)

### P2-1 | Flow Wizard > Step 2 "max 5 photos" non enforced côté client
**Flow :** `/dashboard/new` — step 2 photos
**Scénario :** L'utilisateur sélectionne 8 photos d'un coup dans le file picker (multiple).
**Observation :** `handlePhotoUpload` itère sur `Array.from(e.target.files)` sans limite. L'UI indique "max 5 photos" mais rien ne bloque. 8 uploads simultanés vers `/api/upload` peuvent déclencher le rate limit ou saturer la connexion mobile.
**Fix :** `if (uploadedPhotos.length + files.length > 5) { setUploadError('Maximum 5 photos'); return }`

### P2-2 | Flow Wizard > Upload vidéo affiche une erreur mais l'onglet "Upload" reste actif
**Flow :** `/dashboard/new` — step 3 vidéos UGC
**Scénario :** L'utilisateur clique sur la zone d'upload vidéo.
**Observation :** `handleVideoUpload` appelle `setUploadError(...)` pour expliquer que l'upload vidéo est désactivé. Mais `uploadError` est le même state que les erreurs d'upload photo (partagé). Si l'user a uploadé des photos avec succès et revient à l'étape vidéo, le state `uploadError` peut afficher une ancienne erreur photo, ou vice-versa.
**Fix :** État `videoUploadError` séparé, ou réinitialiser `uploadError` à `null` au changement d'étape.

### P2-3 | Flow Cancellation Stripe > Quota non remis à zéro immédiatement
**Flow :** Annulation subscription → webhook `customer.subscription.deleted`
**Scénario :** User Pro annule → webhook → `plan: 'free'`, `pages_used_this_month: 0`. Mais si l'user avait déjà utilisé 280 pages ce mois (plan Pro = 300), le reset à 0 est correct. Cependant si l'annulation a effet en fin de période (`cancel_at_period_end`), Stripe envoie d'abord `customer.subscription.updated` (status: `active`, cancel_at_period_end: true), puis `customer.subscription.deleted` à l'échéance.
**Observation :** Le webhook `customer.subscription.updated` ne gère pas `cancel_at_period_end: true` — il met à jour le plan normalement (plan reste Pro). C'est correct fonctionnellement, mais l'UI settings ne reflète pas l'état "annulation programmée".
**Fix :** Dans `customer.subscription.updated`, détecter `sub.cancel_at_period_end === true` et l'afficher dans l'UI (ex: badge "Annulation le [date]" dans `/dashboard/settings`).

### P2-4 | Flow Cron > Aucune limite de batch — 500 previews en un seul fetch
**Flow :** `/api/cron/preview-emails`
**Scénario :** À l'échelle (10k users), le cron fetch 500 previews et les itère séquentiellement. Chaque itération fait 2 requêtes Supabase (update + sendEmail).
**Observation :** `.limit(500)` protège contre un OOM, mais 500 × 2 opérations séquentielles dans une Vercel Serverless Function (timeout 60s) peut timeouterr. Il n'y a pas de pagination ni de chunking par lot de 50.
**Fix :** Chunker en batches de 50 avec `Promise.allSettled` par batch, ou utiliser une queue (BullMQ, Trigger.dev).

### P2-5 | Flow Preview > Countdown timer rafraîchi toutes les 60s mais affiche `Xj Yh Zm`
**Flow :** `/preview/[id]` — `useCountdown`
**Scénario :** Si la preview expire pendant que l'utilisateur la regarde, le timer affiche "Expirée" mais l'iframe continue d'afficher le contenu (pas de reload/redirect).
**Observation :** Le timer (ligne 27 : `setTimeLeft('Expirée')`) n'a pas d'effet de bord. L'iframe continue. Pas bloquant (la preview est visible même expirée côté client), mais incohérent : le serveur retournerait 410 si on rechargeait.
**Fix :** Si `diff <= 0`, appeler `router.refresh()` ou recharger les data depuis l'API pour afficher l'écran "expiré".

### P2-6 | Flow Dashboard > Pas de empty state sur la liste de pages
**Flow :** `/dashboard/pages`
**Scénario :** Nouvel utilisateur qui vient de s'inscrire, 0 page créée.
**Observation :** Non audité directement (PagesClient.tsx non lu en détail). À vérifier : existe-t-il un empty state avec CTA "Créer ma première page" ?
**Recommandation :** Ajouter un empty state illustré si `pages.length === 0` avec lien vers `/dashboard/new`.

### P2-7 | Flow Auth > `tokenValid === null` (état initial) affiche rien
**Flow :** `/signup` — validation token en cours
**Scénario :** Entre le chargement de la page et la résolution du fetch `/api/invitations/validate`, `tokenValid` est `null`.
**Observation :** L'état `validating = true` affiche un loader, mais si `setValidating(false)` est appelé avant que `setTokenValid` soit défini (race condition sur `finally` vs `then`), l'état peut momentanément avoir `tokenValid = null` et `validating = false`. Dans ce cas, aucune des branches (`if (validating)`, `if (tokenValid === false)`, `if (tokenValid === null)`) ne matche, et le composant retourne `undefined` / render vide.
**Fix :** Ajouter `if (tokenValid === null) return null` (ou un loader) pour couvrir l'état intermédiaire avant que la validation commence.

### P2-8 | Flow i18n > Wizard accepte `zh` et `ar` mais l'UI est full FR
**Flow :** `/dashboard/new` — step 7 langue
**Scénario :** L'utilisateur sélectionne `ar` (arabe) ou `zh` (chinois).
**Observation :** La page générée aura du contenu en arabe ou chinois (validé par `ALLOWED_LANGS`). Mais l'interface du wizard est entièrement en français. Les labels de step, les messages d'erreur, les toasts, les placeholders sont tous en FR. L'user arabophone qui génère en AR navigue dans une interface incompréhensible.
**De plus :** `ar` est RTL. Si le contenu généré est RTL mais l'iframe preview ne définit pas `dir="rtl"` sur le `<html>`, la mise en page peut être cassée dans le template (les templates n'ont pas été audités pour le support RTL).
**Fix :** Documenter clairement que le wizard est FR-only (pour le beta). Pour `ar` : valider que `<html lang="ar" dir="rtl">` est posé par le template renderer.

### P2-9 | Flow Mobile > StepIndicator masqué sur mobile (`hidden sm:flex`)
**Flow :** `/dashboard/new` — wizard mobile
**Scénario :** Un utilisateur sur iPhone utilise le wizard.
**Observation :** `StepIndicator` (ligne 113) a `className="hidden sm:flex"`. Il est complètement invisible en mobile. La progress bar textuelle ("Étape 3/8 — Vidéos") reste, mais l'indicateur visuel des étapes n'existe pas sous `640px`. Les boutons Précédent/Suivant ont des touch targets de `py-2.5` (environ 42px) — juste sous le minimum recommandé de 44px.
**Fix :** Sur mobile, remplacer le `StepIndicator` horizontal par une version compacte verticale, ou conserver au minimum des points cliquables. Passer les boutons nav à `py-3` minimum.

### P2-10 | Flow Stores > Suppression store sans vérifier les pages publiées sur ce store
**Flow :** `/dashboard/stores` — suppression d'un store
**Scénario :** L'utilisateur supprime un store Shopify qui avait des pages publiées via KONVERT.
**Observation :** La ConfirmModal avertit de la suppression mais ne mentionne pas les pages actives sur ce store. Après suppression, les pages qui avaient été poussées sur Shopify continuent d'exister sur Shopify (pas de rollback). L'entrée en DB du store est supprimée. Les pages en DB référençant ce store_id deviennent orphelines (aucun FK sur `pages.store_id` visible dans l'UI).
**Fix :** Avant suppression, compter les pages publiées sur ce store et avertir : "Ce store a X page(s) publiée(s). Les pages resteront actives sur Shopify mais ne seront plus gérables depuis KONVERT."

### P2-11 | Flow Error > `publishError` toast fermé uniquement au clic — aucun auto-dismiss
**Flow :** `/dashboard/new` — modal publication
**Observation :** Le toast `publishError` (ligne 706-714) se ferme uniquement au `onClick` sur le div. Il n'y a pas de `setTimeout` auto-dismiss. Si l'user ne clique pas, l'erreur rouge reste visible indéfiniment, même après une publication réussie sur un autre store (les deux toasts coexistent).
**Fix :** `useEffect(() => { if (publishError) { const t = setTimeout(() => setPublishError(null), 5000); return () => clearTimeout(t) }}, [publishError])`

### P2-12 | Flow A11y > Focus states sur inputs : `outline-none` omniprésent (42 occurrences)
**Observation :** 42 occurrences de `outline-none` dans le code TSX. La majorité des inputs et boutons supprime l'outline natif. Certains ont des compensations inline via `onFocus/onBlur` qui changent `borderColor`, mais ces compensations ne sont pas détectées par les outils de navigation clavier (elles s'appliquent au border, pas à l'outline). Les inputs du wizard et du formulaire signup ont `onFocus` → border violet : acceptable mais non standard. Les boutons icône (X, upload) sans border n'ont aucune indication de focus.
**Fix :** Remplacer `outline-none` par `focus:outline-none focus-visible:ring-2 focus-visible:ring-violet-500` ou équivalent Tailwind. Garder `outline-none` uniquement pour les éléments ayant une vraie compensation focus.

## 10 cas E2E Playwright — Smoke tests prioritaires avant lancement

Ces 10 tests couvrent les golden paths et les edge cases P0/P1 les plus critiques. Durée estimée d'exécution : 8-12 minutes.

### TC-01 | Tunnel essai — golden path mode manuel
