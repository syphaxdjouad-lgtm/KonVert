## P0 — Bloquants

### P0-1 — `generate/public/route.ts` : aucune vérification anti-hallucination ni `cleanProduct`
**Fichier** : `src/app/api/generate/public/route.ts:64-70`  
**Description** : Quand `url` est fournie, le résultat de `scrapeProduct()` est passé directement à `generateLandingPage()` sans passer par `cleanProduct()` ni `looksHallucinated()`. La route authentifiée `/api/generate` effectue ces deux contrôles (lignes 61, 91-101) ; la route publique les saute complètement. Un produit avec titre générique "Women Summer Dress" ou aucune image valide génère quand même une page IA et la sauvegarde en base.  
**Fix** : Ajouter `product = cleanProduct(raw)` après le scrape, puis appeler `looksHallucinated(product)` et retourner 422 si fake, identique à la logique de la route authentifiée.

### P0-2 — `generate/public/route.ts` : `productInput` accepté sans validation ni nettoyage
**Fichier** : `src/app/api/generate/public/route.ts:66-67`  
**Description** : Si le body contient `product` (saisie manuelle de l'utilisateur anonyme), il est passé tel quel à `generateLandingPage()` sans `cleanProduct()`. Un attaquant peut envoyer un objet arbitraire avec des champs XSS (balises HTML dans le titre, URLs d'images arbitraires non-HTTP) qui seront sauvegardés dans `html_content` puis servis à tous les visiteurs de `/preview/[id]`.  
**Fix** : Appliquer `cleanProduct()` sur `productInput` comme le fait la route authentifiée (ligne 114 de `/api/generate/route.ts`).

### P0-3 — `api/ab/route.ts` : création des variantes sans vérification d'erreur
**Fichier** : `src/app/api/ab/route.ts:205-216`  
**Description** : Les deux `await supabaseAdmin.from('ab_variants').insert(...)` ignorent leurs valeurs de retour. Si l'insert de la variante A échoue (contrainte DB, schema mismatch), le test A/B est créé en base sans aucune variante — le GET `/api/ab` retournera ensuite `Variante introuvable` pour tous les visiteurs de la page, cassant silencieusement le test sans aucune alerte.  
**Fix** : Destructurer `{ error }` sur chaque insert et retourner 500 + rollback du test si une erreur survient.

### P0-4 — `api/email/unsubscribe/route.ts` : désabonnement preview non géré
**Fichier** : `src/app/api/email/unsubscribe/route.ts:10-25`  
**Description** : La fonction `unsubscribeEmail()` met `converted: true` sur `public_previews` (ce qui coupe la séquence preview) **mais n'écrit pas** `emails_sent: [-1]` sur cette table. Or le cron `preview-emails` vérifie `emailsSent.includes(-1)` sur le champ `emails_sent` de `public_previews` (ligne 81). Un utilisateur qui se désabonne via le lien email continuera à recevoir les emails J+1, J+3, J+5, J+7 de la séquence preview.  
**Fix** : Ajouter `.update({ emails_sent: [-1] }).eq('email', normalizedEmail)` sur `public_previews` dans `unsubscribeEmail()`, en plus de l'update existant sur `users`.

## P1 — Majeurs

### P1-1 — `stripe/webhook/route.ts:32` : cast `as unknown` contourne les types Stripe
**Fichier** : `src/app/api/stripe/webhook/route.ts:32`  
**Description** : `extractPeriodEnd()` caste `sub as unknown as { current_period_end?: number }` alors que `Stripe.Subscription` expose `current_period_end` nativement dans les types SDK récents. Ce double cast masque un potentiel undefined : si la propriété n'existe pas (abonnements Stripe sans billing period), la fonction retourne `0`, ce qui écrit `new Date(0 * 1000)` = `1970-01-01` comme `current_period_end` en base — l'abonnement apparaît immédiatement expiré.  
**Fix** : Utiliser `sub.current_period_end ?? 0` directement (disponible dans `@stripe/stripe-js` récent) ou logger explicitement si la valeur est 0/undefined.

### P1-2 — `agency/page.tsx:20` : `loadWorkspaces` appelée avant sa déclaration (TDZ)
**Fichier** : `src/app/(dashboard)/dashboard/agency/page.tsx:20`  
**Description** : ESLint confirme `react-hooks/immutability` — `loadWorkspaces` est appelée dans `useEffect` ligne 20 alors qu'elle est déclarée ligne 30. En pratique le moteur JS hisse les déclarations de fonctions (`function` keyword), donc ça fonctionne aujourd'hui. Mais si la fonction est un jour convertie en `const loadWorkspaces = async () =>`, le code crashe avec une `ReferenceError` en runtime sans avertissement TS. Le linter le signale comme une erreur (pas un warning).  
**Fix** : Déplacer la déclaration de `loadWorkspaces` avant le `useEffect`, ou la passer en `useCallback`.

### P1-3 — `dashboard/agency/page.tsx` : `fetch('/api/workspaces')` sans gestion de réseau down
**Fichier** : `src/app/(dashboard)/dashboard/agency/page.tsx:31-36`  
**Description** : `loadWorkspaces()` fait un `await fetch(...)` sans `try/catch`. Si le réseau est coupé, la promesse rejette et l'exception est non catchée — la page reste en `loading: true` indéfiniment sans message d'erreur à l'utilisateur.  
**Fix** : Envelopper dans `try/catch`, appeler `setLoading(false)` dans `finally` et exposer l'erreur réseau via `setError`.

### P1-4 — `analytics/page.tsx:27-28` : `Date.now()` appelé pendant le render (Server Component)
**Fichier** : `src/app/(dashboard)/dashboard/analytics/page.tsx:27-28`  
**Description** : ESLint React Compiler signale `Cannot call impure function during render`. `Date.now()` est appelé directement dans le corps du composant Server. En pratique, un Server Component Next.js n'est pas un composant React pur au sens du compiler, mais la date est figée au moment du premier render serveur — elle ne se met jamais à jour. Si la page est mise en cache (ce qui est le cas sur Vercel avec ISR), `since30` et `since7` pointent vers une fenêtre de 30/7 jours qui ne glisse plus.  
**Fix** : Ajouter `export const dynamic = 'force-dynamic'` sur la page analytics pour garantir un render frais à chaque requête, ou calculer les dates dans un Server Action.

### P1-5 — `KonvertEasterEgg.tsx:58-62` : `Math.random()` pendant le render → hydration mismatch
**Fichier** : `src/components/marketing/KonvertEasterEgg.tsx:58-62`  
**Description** : Quatre appels `Math.random()` sont effectués dans le corps du composant pour positionner les confettis. Si ce composant est rendu côté serveur (même partiellement), les valeurs aléatoires seront différentes entre le HTML serveur et le render client — hydration mismatch garanti, pouvant provoquer un re-render complet de la page et des erreurs console en prod.  
**Fix** : Envelopper les valeurs dans un `useMemo` ou `useState` avec initialisation côté client uniquement, ou ajouter `'use client'` + initialiser via `useEffect`.

### P1-6 — `signup/page.tsx:24` : `setTokenValid(false)` appelé synchronement dans `useEffect`
**Fichier** : `src/app/(auth)/signup/page.tsx:24`  
**Description** : ESLint `react-hooks/set-state-in-effect` le confirme. L'appel synchrone de `setState` dans un effet sans condition asynchrone déclenche un double render immédiat inutile. En React 18 Concurrent Mode, cela peut provoquer des artifacts visuels sur la page de signup : un flash d'état "token invalide" avant le fetch de validation.  
**Fix** : Remplacer par un state initial `null` et évaluer `!token` dans le render plutôt que dans l'effet : `useEffect(() => { if (!token) return; ... }, [token])`.

### P1-7 — `shopify/push/route.ts:83-90` : message d'erreur interne exposé en prod
**Fichier** : `src/app/api/shopify/push/route.ts:83-90`  
**Description** : Le catch final retourne `{ error: err instanceof Error ? err.message : 'Erreur Shopify' }`. Le message brut de l'exception Shopify peut contenir des détails internes (endpoint, token partiel, stack trace SDK). C'est identique pour `woocommerce/push` et `youcan/push`. En comparaison, `api/generate` a un filtre explicite qui ne retourne que des messages génériques.  
**Fix** : Normaliser les messages d'erreur : retourner uniquement le message d'erreur Shopify/WooCommerce/YouCan (déjà formaté par les clients lib) mais logguer l'erreur complète côté Sentry.

### P1-8 — `ab-test/page.tsx:75` : `router` et `supabase` absents des dépendances `useCallback`
**Fichier** : `src/app/(dashboard)/dashboard/pages/[id]/ab-test/page.tsx:75`  
**Description** : ESLint `react-hooks/exhaustive-deps` le confirme : `router` et `supabase` sont utilisés dans `fetchData` mais absents du tableau de dépendances `[pageId]`. Le compiler React a même skipé l'optimisation de cette callback. Si `router` change (navigation), `fetchData` garde une référence stale — un appel à `router.push('/login')` dedans pourrait ne pas déclencher la navigation.  
**Fix** : Ajouter `router` et `supabase` dans les dépendances, ou extraire `supabase` à l'extérieur du composant (instance singleton client).

### P1-9 — `woocommerce/connect/route.ts:53` : auth vérifiée APRÈS l'appel réseau externe
**Fichier** : `src/app/api/woocommerce/connect/route.ts:38-53`  
**Description** : Le client WooCommerce est instancié et `client.ping()` est appelé (requête HTTP vers le store externe) **avant** de vérifier l'authentification Supabase (ligne 53). N'importe qui peut donc déclencher une requête HTTP arbitraire vers n'importe quel endpoint WooCommerce accessible depuis les serveurs Vercel sans être authentifié. Le guard SSRF bloque les IPs privées mais pas les endpoints publics.  
**Fix** : Monter le check `createClient()` + `supabase.auth.getUser()` en tête de route, avant l'instanciation du client WooCommerce.

### P1-10 — `api/generate/route.ts:114` : `cleanProduct(body.product)` sans validation de type
**Fichier** : `src/app/api/generate/route.ts:114`  
**Description** : `body.product` est le résultat de `req.json()` — il peut contenir n'importe quel type. `cleanProduct()` traite `product as ScrapedProduct & {...}` avec des `?? ''` défensifs, mais si `body.product` est `null`, `undefined`, un array, ou un string, les accès `product.images`, `product.variants` etc. peuvent produire des comportements inattendus (TypeScript ne valide pas le body JSON à runtime).  
**Fix** : Ajouter une guard `if (typeof body.product !== 'object' || body.product === null || Array.isArray(body.product))` avant `cleanProduct()`.

### P1-11 — `api/workspaces/[id]/report/route.ts:82-189` : paramètre `any` dans les fonctions de build
**Fichier** : `src/app/api/workspaces/[id]/report/route.ts:82`  
**Description** : `buildReportHtml({ workspace, pages, stats, generatedAt }: any)` et `buildReportPdf({ ... }: any)`. Ces fonctions reçoivent les données workspace incluant `brand_color` et `client_name` qui sont des données utilisateur. Sans type strict, un champ manquant (ex: `pages` undefined au lieu d'array vide) crashe silencieusement en `.map()` — la route retourne 500 sans message clair.  
**Fix** : Typer le paramètre avec une interface dédiée ; `buildReportPdf` en particulier ne retourne pas son résultat typé (ligne 44 : `const pdfBytes = buildReportPdf(reportData)` mais la fonction n'est pas lue dans ce fichier — vérifier qu'elle existe).

### P1-12 — `api/stripe/checkout/route.ts:11` : cast JSON sans validation
**Fichier** : `src/app/api/stripe/checkout/route.ts:11`  
**Description** : `const { plan } = await req.json() as { plan: PaidPlan }` — le cast TypeScript est purement compile-time. La validation ligne 13 `if (!['starter', 'pro', 'agency'].includes(plan))` est correcte mais arrive après le cast. Si `req.json()` échoue (body malformé), la promesse rejetée remonte directement dans le catch sans message spécifique. Surtout : si `plan` est `undefined`, `includes(undefined)` retourne `false` correctement, mais le message d'erreur "Plan invalide" ne distingue pas un body vide d'un plan inconnu.  
**Fix** : Ajouter `req.json().catch(() => ({}))` comme dans `/api/generate/route.ts:14` pour éviter le crash silencieux.

### P1-13 — `notifications/route.ts:40-41` : `select('*')` sur tables sensibles sans projection
**Fichier** : `src/app/api/notifications/route.ts:40-41`  
**Description** : `.select('*')` sur `users` et `subscriptions` charge l'intégralité des colonnes incluant `access_token` chiffré (si présent sur users), `stripe_customer_id`, `stripe_subscription_id`. Ces données ne sont jamais utilisées dans la génération des notifications (seuls `plan`, `pages_used_this_month`, `created_at` et `current_period_end` sont lus). Le select large augmente la surface d'exposition en mémoire et ralentit la requête inutilement.  
**Fix** : Remplacer par des projections explicites : `.select('plan, pages_used_this_month, created_at')` sur users, `.select('status, current_period_end')` sur subscriptions.

### P1-14 — `dashboard/layout.tsx:44` : `setMobileMenuOpen(false)` dans useEffect sans valeur stable
**Fichier** : `src/app/(dashboard)/layout.tsx:43-45`  
**Description** : ESLint le signale. Appeler `setState` synchronement dans un effet sur `pathname` provoque un render cascade : premier render avec `pathname` changé → effet → setState → second render. En React 18 Strict Mode (activé en dev Next.js), cela triple les renders. En prod c'est simplement un pattern sous-optimal qui peut créer des artifacts visuels au changement de page.  
**Fix** : Ce cas spécifique (fermer le menu au changement de route) est l'un des rares où `setState` dans un effet est acceptable selon les docs React. Ajouter un commentaire `// eslint-disable-next-line react-hooks/set-state-in-effect` avec justification, ou utiliser un ref booléen pour éviter le render superflu.

### P1-15 — `cron/trial-emails` : `dayToSend` prend **le premier jour passé non envoyé** au lieu du dernier
**Fichier** : `src/app/api/cron/trial-emails/route.ts:79-81`  
**Description** : `TRIAL_DAYS.find((d) => d <= daysPassed && !emailsSent.includes(d))` retourne le **premier** jour de la liste (`TRIAL_DAYS = [1,3,7,10,12,13,14]`) qui satisfait la condition. Si un utilisateur s'inscrit et que le cron ne tourne pas pendant 10 jours (downtime, CRON_SECRET manquant), le premier appel envoie l'email J+1 au lieu de J+7 ou J+10. Le comportement souhaité est probablement d'envoyer le mail du jour le plus récent non envoyé.  
**Fix** : Remplacer `find` par `findLast` (ES2023, disponible en Node 18+) pour envoyer l'email du jour le plus avancé non encore envoyé : `TRIAL_DAYS.findLast((d) => d <= daysPassed && !emailsSent.includes(d))`.

### P1-16 — `api/scrape/route.ts` : erreur interne Firecrawl exposée dans la réponse 500
**Fichier** : `src/app/api/scrape/route.ts:57-65`  
**Description** : En cas d'échec, la route retourne `{ error: "Scraping échoué : [msg]", debug: detail }` avec `detail` contenant potentiellement des informations internes (URLs internes Firecrawl, clés API partielles dans les messages d'erreur HTTP). Ce champ `debug` est exposé directement dans la réponse JSON publique.  
**Fix** : Logguer `detail` côté Sentry uniquement, retourner au client un message générique sans le champ `debug`.

### P2-18 — `api/generate/public/route.ts:126-139` : fire-and-forget email sans log d'erreur
**Fichier** : `src/app/api/generate/public/route.ts:126-139`  
**Description** : `fetch(...).catch(() => {})` — l'email J+0 est envoyé en fire-and-forget avec un catch vide. **Fix** : Remplacer `.catch(() => {})` par `.catch((err) => { console.error('[generate/public] email J+0 failed:', err) })` au minimum, ou tracker via Sentry.
